products (1) ──⟶ sales (many)    FK = product_id → products.id
CREATE TABLE products ( id INTEGER PRIMARY KEY AUTOINCREMENT, name TEXT, price REAL, stock INTEGER ); CREATE TABLE sales ( id INTEGER PRIMARY KEY AUTOINCREMENT, product_id INTEGER, qty INTEGER, sale_date TEXT, FOREIGN KEY (product_id) REFERENCES products(id) );

residents: id (PK, INTEGER), full_name (TEXT), address (TEXT), birth_date (TEXT).
clearances: id (PK, INTEGER), resident_id (FK → residents.id), purpose (TEXT), date_issued (TEXT), status (TEXT).
The FK resident_id is on clearances (the many side). One resident → many clearances.

🔧 Exercise 2: Fix the one-giant-table

❌ Bad: a single records table crams product info into every sale:
sale_id · product_name · product_price · qty · date
(product_name & price are repeated on every sale — if the price changes, you must update dozens of rows.)

Split this into two properly-related tables. Write the two table names and the foreign key that connects them.

Check your answer
Table 1 — products: id (PK), name, price.
Table 2 — sales: id (PK), product_id (FK → products.id), qty, date.
Now product name/price live in ONE place. Sales just store the product's id. ✅

Memory trick: "The many side carries the id of the one." Sales carry product_id. Clearances carry resident_id.
